agent-webhook-bridge · docs

Instalación y uso

Un broker HTTP standalone que recibe webhooks externos (CI, GitHub, Flowise…) y despierta a Claude Code — como sesión nueva o reanudando una existente — sin que Claude tenga que estar corriendo de antemano.

Requisitos

  • Node.js 24 o superior. El broker usa node:sqlite (sin dependencias nativas) y corre los archivos .ts directamente — no hay paso de build.
  • Claude Code CLI instalado y autenticado, con el binario claude en el PATH de la máquina donde corre el broker.

Instalación

No hay publicación en npm todavía: se usa directo desde el checkout del repo.

cd agent-webhook-bridge
npm install        # opcional — solo trae @types/node para el editor,
                    # no hace falta para ejecutar nada

Para tener el comando awb disponible en cualquier carpeta:

npm link           # crea el symlink global "awb" -> ./cli/awb.ts

Si preferís no linkearlo globalmente, cualquier comando awb … de esta guía funciona igual como node cli/awb.ts … ejecutado desde la carpeta del repo.

Arrancar el broker

awb start

Corre en primer plano y escucha en 127.0.0.1:8890 por defecto (puerto elegido a propósito para no chocar con el webhook-receiver de free-code, que usa 8787-8806).

config

La configuración (hooks registrados) vive en ~/.agent-webhook-bridge/hooks.json, la cola de eventos en ~/.agent-webhook-bridge/events.db (SQLite) y los logs de cada invocación de Claude en ~/.agent-webhook-bridge/logs/.

Para usar otra ubicación (por ejemplo en tests, o para correr varias instancias), seteá AWB_HOME=/ruta/propia antes de cualquier comando awb.

El broker relee hooks.json en cada request, así que podés registrar o borrar hooks con awb add/awb rm mientras sigue corriendo, sin reiniciarlo.

Registrar un hook

awb add ci-failures \
  --trigger \
  --workdir /ruta/a/tu/repo \
  --prompt-template 'Un build falló en CI. Log:\n\n{{payload}}\n\nInvestigá la causa.'

La salida devuelve la URL local y el secreto que hay que configurar en el sistema externo:

Hook 'ci-failures' [trigger] consumers=spawn:claude
Local URL:  http://127.0.0.1:8890/hook/ci-failures
Header:     X-Webhook-Secret: 76171625f1dbab4647623cb7f6c99e18...
Workdir:    /ruta/a/tu/repo
Prompt:     Un build falló en CI. Log:\n\n{{payload}}\n\nInvestigá la causa.
OpciónQué hace
--trigger / --queueModo de entrega. trigger (default) dispara spawn:claude al llegar el evento. queue solo lo persiste en SQLite, para un futuro adaptador de lectura (ver Limitaciones).
--consumer <c>Repetible. Default: spawn:claude si es trigger, queue si es queue.
--prompt-template <texto>Plantilla del prompt que recibe Claude. {{payload}} se reemplaza por el body del evento (formateado como JSON si no es texto plano); {{hook}} por el nombre del hook.
--workdir <dir>cwd del proceso claude spawneado. También es la clave que serializa ejecuciones: dos eventos con el mismo workdir nunca corren claude en paralelo sobre el mismo repo.
--secret <s> / --hmac-secret <s>Autenticación. Si no das ninguna, se genera un secreto aleatorio. Con --hmac-secret el caller firma el body crudo en vez de mandar el secreto en un header.
--permission-mode <modo>Se pasa tal cual como --permission-mode de claude (acceptEdits, auto, bypassPermissions, manual, dontAsk, plan). Sin esto, las corridas headless no pueden escribir ni editar nada (ver Permisos en modo headless).

El body del curl es tuyo, no un formato fijo

El bridge no espera ningún campo en particular en el body — ni log, ni branch, ni nada. Vos elegís qué manda tu sistema externo (o qué escribís a mano para probar); el broker se limita a agarrar ese body tal cual llegó y pegarlo entero donde diga {{payload}} en tu --prompt-template. No hay sustitución campo por campo — es un único bloque de texto.

Con el hook de arriba, esta request:

curl -X POST http://127.0.0.1:8890/hook/ci-failures \
  -H "Content-Type: application/json" \
  -H "X-Webhook-Secret: <secreto>" \
  -d '{"branch":"main","step":"npm test","exitCode":1}'

termina armando este prompt exacto, que es lo que claude -p recibe como argumento:

Un build falló en CI. Log:

{
  "branch": "main",
  "step": "npm test",
  "exitCode": 1
}

Investigá la causa.

Claude recibe ese JSON como texto plano dentro del prompt y lo interpreta él mismo — no hay ningún parseo especial de branch/step/exitCode del lado del broker. Por eso en distintos ejemplos de esta guía el body tiene forma distinta ({"log": "…"}, {"flow": "…"}, {"branch": "…"}): no es que cada uno siga un formato requerido, es que cada uno simula lo que mandaría un sistema distinto (un test runner, un flujo de Flowise, un paso de CI). Vos decidís la forma del body de tu propio caller, y redactás el --prompt-template para que tenga sentido con esa forma.

tip

Si el Content-Type no es application/json o el body no es JSON válido, {{payload}} se reemplaza igual, pero con el texto crudo tal cual — también funciona mandar texto plano en -d, no hace falta que sea JSON.

Sesión nueva vs. reanudar (sessionId)

Para hooks con consumer spawn:claude, el adaptador decide cómo lanzar Claude según un único header en la request entrante:

POST /hook/<nombre> header sessionId presente? │ ├─ no → claude -p "<prompt>" --output-format json │ (arranca una sesión nueva) │ └─ sí → claude --resume <sessionId> -p "<prompt>" --output-format json (reanuda esa sesión existente, con todo su contexto previo)

Para que --resume encuentre la sesión, el hook tiene que tener el --workdir igual al directorio del proyecto donde esa sesión se creó originalmente — las sesiones de Claude Code están asociadas a un directorio de trabajo.

detalle no obvio

--resume <id> sin -p/--print abre el picker interactivo de Claude Code. En un proceso spawneado sin TTY eso se queda colgado para siempre — por eso el adaptador siempre agrega -p también en la rama de resume, no solo en la de sesión nueva.

¿De dónde saco el sessionId que tengo que mandar?

No lo inventa el broker ni Claude: es el caller (tu script, tu flujo de Flowise, tu pipeline de CI) el que tiene que conocerlo y mandarlo como header HTTP literal sessionId: <uuid> en el POST. Tres formas de conseguir ese UUID:

  1. Del propio Claude, en la primera respuesta. Toda invocación con --output-format json devuelve un campo session_id en el JSON de salida — incluida la que arranca claude -p sin resume. Guardalo la primera vez y reusalo en las próximas llamadas al mismo hook para seguir esa misma conversación.
  2. De una sesión interactiva que ya tenés abierta. El id es el nombre del archivo <sessionId>.jsonl dentro de ~/.claude/projects/<tu-proyecto-codificado>/ (una carpeta por proyecto, con guiones en vez de /). También lo ves en el picker de /resume dentro de Claude Code.
  3. De un log del broker. Cada invocación queda registrada en ~/.agent-webhook-bridge/logs/<hook>-<timestamp>.log, con la respuesta completa de Claude — ahí también está el session_id usado.

Ejemplo de capturar el id en la primera llamada y reusarlo en la segunda:

# 1) primera llamada: sin sessionId, arranca una sesión nueva.
#    -w guarda el body de la respuesta del broker, pero el session_id real
#    vive en el log de la invocación de Claude (el broker solo confirma "ok").
curl -s -X POST http://127.0.0.1:8890/hook/ci-failures \
  -H "Content-Type: application/json" \
  -H "X-Webhook-Secret: <tu secreto>" \
  -d '{"log":"npm test exit 1"}'

# 2) leé el session_id de esa corrida en el log más reciente del hook
SID=$(ls -t ~/.agent-webhook-bridge/logs/ci-failures-*.log | head -1 \
  | xargs cat | grep -o '"session_id":"[^"]*"' | cut -d'"' -f4)
echo "session capturada: $SID"

# 3) segunda llamada al mismo hook, ahora reanudando esa sesión
curl -s -X POST http://127.0.0.1:8890/hook/ci-failures \
  -H "Content-Type: application/json" \
  -H "X-Webhook-Secret: <tu secreto>" \
  -H "sessionId: $SID" \
  -d '{"log":"el mismo build sigue fallando, segundo intento"}'

Permisos en modo headless

Una corrida spawneada no tiene terminal: si el prompt le pide a Claude escribir o editar un archivo, no hay nadie que apruebe ese permiso, así que por defecto la acción queda denegada y Claude responde explicando que no pudo hacerlo. Esto pasa incluso reanudando una sesión que interactivamente ya venías usando sin problema — el permiso no "se hereda" a una corrida headless nueva.

Log real de un hook sin --permission-mode pidiéndole que guarde un archivo:

"result": "Sigo sin permiso para escribir el archivo -- el sistema requiere que
alguien apruebe la acción de escritura desde la sesión interactiva ...",
"permission_denials": [{ "tool_name": "Write", "tool_input": { "file_path": "..." } }]

La solución es declarar --permission-mode al registrar el hook:

awb rm resume
awb add resume --trigger \
  --workdir /home/lenovo/Documentos/free-code/free-code \
  --permission-mode acceptEdits

Con eso, el mismo pedido termina escribiendo el archivo de verdad:

"result": "Guardado en `/home/lenovo/Documentos/free-code/free-code/resumen-sesion-flowise-webhook.md` (raíz del working directory)."
ModoQué autoriza
acceptEditsAuto-aprueba Write/Edit de archivos. Es el que conviene para la mayoría de los hooks de automatización — no habilita nada más.
bypassPermissionsSalta todos los chequeos de permiso, incluido Bash arbitrario. Mismo riesgo que --dangerously-skip-permissions: solo tiene sentido en un sandbox aislado, no en tu máquina con tus repos reales.
plan / manual / dontAsk / autoOtros modos de claude; ver claude --help para el detalle de cada uno.
a propósito, no default

permissionMode queda sin setear por defecto en un hook nuevo — tenés que pedirlo explícitamente. La razón: una vez que un hook puede escribir sin preguntar, cualquiera que sepa el secreto del hook puede hacer que Claude escriba archivos en ese workdir con el prompt que quiera. Usalo solo en hooks donde confiás en la fuente del evento y en lo que el --prompt-template le puede llegar a pedir a Claude que haga.

Más ejemplos: registrar y disparar hooks

Trigger simple, con secreto compartido

El caso base ya visto arriba: modo trigger, autenticación por X-Webhook-Secret.

awb add ci-failures --trigger --workdir /home/lenovo/proyectos/mi-repo \
  --prompt-template 'Build falló en CI:\n\n{{payload}}\n\nInvestigá la causa.'

curl -X POST http://127.0.0.1:8890/hook/ci-failures \
  -H "Content-Type: application/json" \
  -H "X-Webhook-Secret: <secreto de awb add>" \
  -d '{"branch":"main","step":"npm test","exitCode":1}'

Queue, para una fuente ruidosa (sin spawnear Claude por cada evento)

Un flujo de Flowise que dispara seguido no debería levantar una sesión de Claude por cada evento — con --queue el evento solo se persiste en SQLite (ver Limitaciones sobre cómo se lee hoy esa cola): 

awb add flowise --queue --consumer queue

curl -X POST http://127.0.0.1:8890/hook/flowise \
  -H "Content-Type: application/json" \
  -H "X-Webhook-Secret: <secreto de awb add>" \
  -d '{"flow":"lead-scoring","result":"qualified","leadId":"8213"}'

awb events flowise   # confirmar que quedó guardado, status "pending"

Autenticación por HMAC en vez de secreto compartido

Con --hmac-secret, en vez de mandar el secreto en un header lo usás para firmar el body crudo y mandás la firma en X-Signature: sha256=<hex>: 

awb add deploys --trigger --hmac-secret 'un-secreto-largo-y-random' \
  --workdir /home/lenovo/proyectos/mi-repo

BODY='{"env":"prod","status":"failed"}'
SIG=$(printf '%s' "$BODY" | openssl dgst -sha256 -hmac 'un-secreto-largo-y-random' | sed 's/^.* //')

curl -X POST http://127.0.0.1:8890/hook/deploys \
  -H "Content-Type: application/json" \
  -H "X-Signature: sha256=$SIG" \
  -d "$BODY"
sin --prompt-template

A este awb add no le pasé --prompt-template a propósito, para mostrar qué pasa si lo omitís: el broker usa la plantilla por defecto "Incoming webhook event for '{{hook}}':\n\n{{payload}}". Con el body de arriba, el prompt que recibe Claude termina siendo: 

Incoming webhook event for 'deploys':

{
  "env": "prod",
  "status": "failed"
}

Para un caso real casi siempre conviene tu propio --prompt-template (que le diga a Claude qué investigar o qué hacer), como en los demás ejemplos de esta guía.

ojo con github

GitHub firma sus webhooks con X-Hub-Signature-256, no con X-Signature. Si querés apuntar un webhook de GitHub directo a este broker hoy, hace falta algo en el medio que renombre ese header (o reenviar el evento con un script) — no hay soporte nativo para el nombre de header de GitHub todavía.

Probar rápido con awb test (sin armar el curl a mano)

# sesión nueva (sin sessionId)
awb test ci-failures --body '{"log":"npm test exit 1"}'

# reanuda una sesión existente
awb test ci-failures --body '{"log":"npm test exit 1"}' \
  --session-id d63be319-433a-4ae1-8a1c-a8a978e53d89

El resultado de cada invocación (comando exacto, stdout/stderr de Claude) queda en ~/.agent-webhook-bridge/logs/<hook>-<timestamp>.log.

Ver estado

awb list             # hooks registrados, modo, URL, secreto
awb url ci-failures  # detalle de un hook puntual
awb events           # últimos eventos recibidos y su estado (pending/delivered/failed)
awb events ci-failures

Seguridad

  • El servidor solo escucha en 127.0.0.1 — no queda expuesto en la red por defecto.
  • La autenticación es obligatoria por hook: un hook sin secret ni hmacSecret configurado nunca autentica ninguna request.
  • Para fuentes que no corren en la misma máquina (webhooks de GitHub, Flowise en otro host), poné el broker detrás de un túnel (cloudflared/ngrok) y preferí hmacSecret sobre secreto compartido plano.
  • Los payloads se guardan tal cual en events.db — si la fuente externa puede incluir tokens o secretos en el body (logs de CI, por ejemplo), tenelo en cuenta antes de compartir esa base de datos.

Limitaciones actuales

Esta es la fase 1 del roadmap (ver PLAN.md en la raíz del repo): broker + adaptador de spawn, solo para Claude Code.

  • El modo queue persiste los eventos en SQLite, pero todavía no existe el adaptador MCP (poll_events/wait_for_event) para leerlos desde una sesión — eso es fase 2.
  • Cursor y Codex CLI no están soportados todavía en el spawn adapter — solo Claude Code.
  • No hay unidad de systemd/launchd ni instalador: awb start corre en primer plano y hay que supervisarlo a mano (o con pm2, tmux, etc.) si querés que sobreviva a un reinicio.
